What is this project?

Carcino Vantage is the internal operating system for The Carcino Foundation. It's not a single tool — it's a multi-department platform where every team has a custom dashboard, and all content flows through a shared editorial pipeline.

01
Markdown Editor
CodeMirror 6 with live preview, tabs, auto-save, and DOCX import
02
Department Dashboards
5 departments, each with a unique dashboard. Role-gated content.
03
Task & Assignment System
Work assignments with statuses, proofreading queues, and approval flows
04
PWA + Push Notifications
Installable app with Serwist service worker and VAPID push
05
Auth System
Email + password with bcrypt, JWT tokens, httpOnly cookies
06
Theming Engine
4 paired dark/light themes applied via CSS variables on <html>
Who reads this doc? New engineers joining the Foundation's dev team, or anyone who needs to understand how the codebase works without spending 3 days reading source files. This guide assumes you know what React, TypeScript, and REST APIs are. If you don't — Google them first.

Tech Stack

Everything the app is built on. Understand this before touching any code.

⚡
Next.js 16
App Router architecture. All pages live in src/app/. API routes are in src/app/api/. Dev mode uses Turbopack (--turbo), production builds use webpack.
🦺
TypeScript
Strict types throughout. Key interfaces live in src/types/index.ts and src/types/task.ts. Don't use any unless you absolutely have to.
🗄️
Supabase
PostgreSQL database. Two clients: supabase (browser, anon key, respects RLS) and supabaseAdmin (server-only, service key, bypasses RLS). Never use the admin client in browser code.
🔐
Custom JWT Auth
NOT Supabase Auth. Custom email+password login with bcrypt hashing and JWT tokens stored in httpOnly cookies (7-day expiry). Sessions also tracked in a login_sessions table.
✏️
CodeMirror 6
The core text editor. Loaded dynamically (no SSR) via @uiw/react-codemirror. Handles Markdown syntax highlighting, keybindings, and the Find & Replace panel.
👁️
react-markdown
Renders the live preview pane. Extended with remark-gfm (tables, task lists, strikethrough), rehype-highlight (code blocks), and rehype-slug (heading IDs for outline sidebar).
🎨
Tailwind CSS
Utility-first styling. But the dashboard and editor also use heavy custom CSS with CSS variables for theming. You'll see both approaches — Tailwind classes AND inline styles with var(--accent) etc.
📦
Serwist (PWA)
Wraps next.config.mjs to generate a service worker. Disabled in dev (Turbopack can't compile it). The service worker lives at src/app/sw.ts and handles push notifications and offline caching.
Other notable dependencies

mammoth — converts .docx files to HTML, which then gets turned into Markdown via turndown + turndown-plugin-gfm. This is the "import Word doc" feature in the editor.

web-push — sends browser push notifications via VAPID. Keys are in push-keys.json and .env. The VAPID private key should never be committed to git.

jose + jsonwebtoken — JWT utilities. jose is the modern one, jsonwebtoken is also present. src/lib/auth.ts uses jsonwebtoken.

react-easy-crop — the avatar cropping modal in the profile settings.

@google-analytics/data — the Marketing dashboard calls Google Analytics 4 Data API to show real-time site metrics. Requires a GCP service account.

googleapis — used alongside the GA4 client for broader Google APIs access.

clsx — utility for conditionally joining CSS class names.

Local Setup

How to get this running on your machine in under 5 minutes (assuming you have Node.js installed).

TERMINAL# 1. Clone or unzip the project
cd project-scribe-main

# 2. Install dependencies (~60 packages)
npm install

# 3. Create your .env.local file (see next section)
cp .env.example .env.local   # if this file exists
# Otherwise create .env.local manually with the vars listed below

# 4. Run the dev server (uses Turbopack)
npm run dev

# Open http://localhost:3000
# The login page appears — you need a user in Supabase to log in
Warning: The service worker (PWA) is disabled in development. This is intentional — Turbopack doesn't support Serwist's webpack plugin. Push notifications only work in production. Don't panic if they're not working locally.

Production Build

TERMINAL# Build uses webpack (not Turbopack) — see package.json
npm run build   # = next build --webpack
npm run start   # serves the built output on :3000
Creating your first user: There's no sign-up page — accounts are created manually in Supabase. Go to your Supabase dashboard → Table Editor → users table → Insert row. The password field must be a bcrypt hash. You can generate one with Node.js:

node -e "const b=require('bcryptjs'); b.hash('yourpassword',10).then(console.log)"

Deployment

The project is designed for Vercel. Zero config — Vercel auto-detects Next.js.

Option A — Vercel CLI
npm install -g vercel
vercel

Follow the prompts. It will ask about framework (Next.js) and then deploy.
Option B — GitHub + Dashboard
Push repo to GitHub → Go to vercel.com/new → Import repo → Framework auto-detected → Add env vars in the Vercel dashboard → Click Deploy.
Cron job: vercel.json schedules a daily push reminder at 02:30 UTC via GET /api/push/remind. This sends task-due-soon push notifications to all users. Vercel Cron runs this automatically on Pro plans.
What gets built at deploy time?

During next build --webpack, Serwist generates public/sw.js (the service worker with precache manifest). This is what makes the PWA work in production.

The Supabase image hostname (wqhsgljvthewgcjjivga.supabase.co) is allowlisted in next.config.mjs for Next.js Image optimization. If you migrate to a different Supabase project, update this hostname.

Auth System

This project uses a completely custom auth system, not Supabase Auth. That's an important distinction — don't confuse the two.

Critical: Supabase Auth is NOT used. Row Level Security (RLS) policies in Supabase should NOT rely on auth.uid() because users are not logged in through Supabase. All API routes use the supabaseAdmin client which bypasses RLS entirely.

Login Flow

User submits
email + password
→
POST /api/auth/login
→
Fetch user
from Supabase
→
bcrypt.compare
password vs hash
→
Sign JWT
(7 days)
→
Set httpOnly
cookie: cw_token

How Routes are Protected

Every API route that needs auth does this exact pattern:

// Every protected API route follows this exact pattern:
const token = req.cookies.get('cw_token')?.value;
if (!token) return NextResponse.json({ error: 'Not authenticated' }, { status: 401 });

const payload = verifyToken(token);  // from lib/auth.ts
if (!payload) return NextResponse.json({ error: 'Invalid token' }, { status: 401 });

// payload = { userId, email, name, adminAccess }
// Now use payload.userId to query only the current user's data

The JWT Payload

JWTPayload interface
userIdstringSupabase row ID of the user
emailstringUser's email address
namestringUser's display name
adminAccessbooleanIf true, can publish content, delete tasks, approve articles, etc.

Client-Side User State

The useUser() hook in src/lib/useUser.ts handles this with a 3-layer caching strategy:

1. In-memory cache
→ fastest, survives hot reloads in dev
2. localStorage cache
→ survives page refresh, shows user instantly
3. GET /api/auth/me
→ always runs in background to keep data fresh

Database (Supabase)

The database is Supabase-hosted PostgreSQL. Here are all the tables you need to know about.

users — the people who log in
users table
iduuidPrimary key
nametextDisplay name
emailtextUnique. Used for login.
passwordtextbcrypt hash. Never store plaintext.
admin_accessbooleanCan publish content, approve articles, assign tasks
departmenttextOne of: "Writers' Block", "Design Lab", "Development", "Marketing", "Leadership"
avatar_urltext?Public URL from Supabase Storage
is_activebooleanIf false, login is blocked with "Account is inactive"
Content tables — blogs, cancer_docs, survivor_stories

There are 3 separate content tables, not one. They all have similar columns but subtle differences:

blogs / cancer_docs / survivor_stories
iduuidPrimary key
title / nametextsurvivor_stories uses name, not title. This trips people up. The editor normalizes it.
slugtextURL-safe identifier. Must be unique per table.
contenttextRaw Markdown content
statustext"draft" | "in_review" | "published". Note: the UI uses "review" but the DB stores "in_review".
author_iduuidFK to users.id
tagstext[]Array of tags. cancer_docs does NOT support tags — omit this column for that table.
wordsintegerWord count (denormalized for dashboard stats)

When the user changes the content type in the editor (e.g. from blogs to cancer_docs), the save API detects the mismatch, inserts into the new table, and deletes from the old one — preserving the document ID.

work_assignments — the task system
work_assignments table
iduuidPrimary key
titletextTask name
descriptiontextFull task description
statustextSee status pipeline below
prioritytext"low" | "normal" | "high"
categorytext"task" | "article" | "blog" | "survivor_story" | "awareness_post"
assigned_touuidSingle assignee FK
assigned_to_idsuuid[]Multi-assignee array. Either this OR assigned_to is used.

push_subscriptions — for browser notifications

Stores browser PushSubscription objects per user. Columns: user_id, endpoint (unique per device), p256dh (encryption key), auth (auth secret). Stale/expired subscriptions are auto-deleted when a push fails with HTTP 410.

Tracks active tokens: user_id, token (the full JWT), expires_at (7 days from login). This table exists for potential session revocation — but the logout route currently just clears the cookie rather than deleting from this table (a minor gap worth noting).

API Routes

Every API endpoint, what it does, and whether you need to be logged in or an admin.

Method Route Auth What it does
POST /api/auth/login None Email+password login. Returns user info + sets cw_token httpOnly cookie.
POST /api/auth/logout Cookie Clears the cw_token cookie.
GET /api/auth/me Cookie Returns the current user's full profile from DB (fresh data, not just JWT payload).
GET /api/documents Cookie Fetches all documents (blogs + survivor_stories + cancer_docs) for the current user. Normalizes and sorts by date.
DELETE /api/documents?id=&type= Cookie Deletes a document. Pass type = table name (blogs/survivor_stories/cancer_docs).
GET /api/editor/load?id=&type= Cookie Loads a single document into the editor. Normalizes the title field (name vs title).
POST /api/editor/save Cookie Creates or updates a document. Handles cross-table migration when content type changes. Syncs status to work_assignments. Admins only can publish.
GET /api/tasks Cookie Returns work_assignments for the current user (or all, if Leadership/admin). Cached 5 minutes via unstable_cache.
PATCH /api/tasks Cookie Updates a task's status, priority, dates, or assignees. Admins can change assignees. Only the task owner can update.
DELETE /api/tasks?id= Admin only Deletes a task. Only the user who created it can delete it, and only if they have admin access.
POST /api/tasks/assign Admin only Creates a new work assignment and optionally links it to a document.
POST /api/upload Cookie Uploads an image or video to Supabase Storage bucket task-submissions. Returns public URL. Allowed types: images + mp4/webm/quicktime.
GET /api/analytics Cookie Fetches GA4 data for last 7 days (active users, page views, bounce rate). Cached 10 minutes. Needs GCP service account env vars.
POST /api/push/subscribe Cookie Saves a browser PushSubscription to Supabase. One record per user+endpoint combo (upserted).
DELETE /api/push/subscribe Cookie Removes a push subscription (unsubscribe).
GET /api/push/remind Cron Runs daily at 02:30 UTC via Vercel Cron. Sends push notifications about upcoming due tasks.
GET /api/content/blogs None Public endpoint. Returns all published blog posts (for the public website).
GET /api/content/blog/[slug] None Public endpoint. Returns a single published blog by slug.
GET /api/github/stats Cookie Returns GitHub repository stats for the Development dashboard widget.
GET /api/team Cookie Returns all active team members for the Team page roster.

The Editor

The editor at /editor is the core content creation tool. It uses CodeMirror 6 as the text engine, supports multiple tabs, and syncs everything to Supabase.

Multi-Tab System
Each tab is a Tab object with its own content, type, slug, and status. Tabs are persisted to localStorage (key: cs-tabs). You can have multiple documents open at once.
Content Types
Each tab has a type: blogs, survivor_stories, or cancer_docs. This determines which Supabase table it saves to. Changing type mid-edit migrates the document.
Auto-Save
Content is saved to localStorage (key: cs-content) every 800ms via debounce. Manual save to Supabase via Ctrl+S or the Save button. Unsaved tabs show a dot indicator.
View Modes
Three modes: editor (CodeMirror only), split (editor + preview side by side), preview (preview only). Controlled by the ViewMode type.
DOCX Import
Drop a .docx file into the editor → mammoth converts it to HTML → turndown converts HTML to Markdown. Special style maps preserve bullet points and numbered lists.
Collaborator Presence
The editor subscribes to a Supabase Realtime channel per document. Other users' cursors broadcast via track() and appear as floating name badges. Uses the browser Supabase client (not admin).

The EditorAPI Interface

The toolbar and keyboard shortcuts communicate with CodeMirror through a stable EditorAPI ref. This keeps the toolbar decoupled from CodeMirror internals:

How it works

// 1. SettingsModal defines theme objects with CSS variable sets
// 2. applySettings() writes them to document.documentElement.style
// 3. ThemeInitializer runs BEFORE React hydrates (in <head>)

Known Gotchas & Tips

Things that will definitely trip you up if you don't know about them.

🚨 survivor_stories uses "name" not "title"

The survivor_stories table has a name column for the document title, while blogs and cancer_docs use title. The API normalizes this, but if you're writing raw Supabase queries, remember to check which column you're selecting.

In the save route: if (table === 'survivor_stories') { data.name = title; } else { data.title = title; }

🚨 cancer_docs does NOT support tags

The tags column only exists on blogs and survivor_stories. The save API explicitly skips setting tags when the table is cancer_docs. If you add this column and forget this check, you'll get a Supabase error on save.

🚨 "in_review" in DB, "review" in UI

The database stores status as in_review, but the editor and UI display it as review. The save route converts: const dbStatus = status === 'review' ? 'in_review' : status. The load route converts back. Don't introduce a third version of this string.

🚨 Never import supabaseAdmin in client components

src/lib/supabaseAdmin.ts uses the SUPABASE_SERVICE_ROLE_KEY which has full database access bypassing Row Level Security. If this key leaks to the browser, anyone can read and write all data. Only import supabaseAdmin in API route files (src/app/api/) or Server Components.

The browser client supabase (from lib/supabase.ts) uses the anon key and is safe for client-side use — but it's mainly used for Realtime (presence) subscriptions in the editor, not data fetching.

🚨 PWA / Service Worker only works in production

The next.config.mjs sets disable: process.env.NODE_ENV !== "production" for Serwist. In npm run dev, there is no service worker, no push notifications, and no offline support. This is expected. To test PWA features, run npm run build && npm run start.

⚠️ unstable_cache for tasks doesn't invalidate on writes

The GET /api/tasks route uses Next.js's unstable_cache with a 5-minute revalidation window. This means after a task is created or updated, the GET response may show stale data for up to 5 minutes. The UI optimistically updates the local state to compensate. If you need real-time task updates, consider removing the cache or using Supabase Realtime.

⚠️ VAPID keys in push-keys.json are for development only

The push-keys.json file commits real VAPID keys for developer convenience. In production, generate fresh keys with node scripts/generate-vapid.js, set them in the Vercel dashboard as env vars, and do NOT use the ones in the repo. Also update the publicKey in push-keys.json to match (since it's served to the client).

⚠️ The Supabase image hostname is hardcoded in next.config.mjs

The images.remotePatterns in next.config.mjs allowlists the specific Supabase project hostname wqhsgljvthewgcjjivga.supabase.co. If you use a different Supabase project, avatar images won't load — you'll get a Next.js image optimization error. Update this hostname to match your project's URL.

💡 EditorPane is dynamically imported (no SSR)

CodeMirror uses browser APIs that don't exist on the server. The editor page uses dynamic(() => import('@/components/EditorPane'), { ssr: false }). This means there's a loading spinner on first render before CodeMirror initializes. This is normal and intentional.

💡 The login page has all its styles inline

The login page uses a massive <style> block inside the component because it predates the global CSS refactor. This is tech debt — the styles could be moved to globals.css. For now, don't be alarmed by the wall of CSS at the top of src/app/login/page.tsx.
